# 출근 마일리지 (Commute Mileage)

## Problem Statement

HMW — 매일 출근하는 직장인에게 환경(미세먼지·강수·풍속·UV·기상특보)을 **적군** 삼아, 출근 자체를 채점하고 영웅 칭호를 부여하는 share-worthy한 미니 RPG로 바꿀 수 있을까?

> 틀 변경: "환경 정보 알려주는 위젯" → "환경을 게임 보스로 바꿔 매일 캡쳐각을 만드는 RPG"

## Recommended Direction

**환경 → 점수 → 칭호의 3-step 자동 파이프라인.**

매일 아침 사용자가 도시(또는 좌표)를 선택하면, Open-Meteo의 weather + air-quality + UV 데이터를 한 번에 가져와 출근 난이도 점수를 계산한다. 그 점수와 환경 컨텍스트를 Gemini Flash로 넘기면 그날의 칭호("미세먼지 검객", "비바람의 기사")와 1줄 영웅 서사가 생성된다. 누적 마일리지는 localStorage에 일자별 점수를 합산해 캘린더 뷰로 시연한다.

이 방향을 고른 이유:
- **AI가 본체** (강의 취지) — 칭호·서사 자유 작문이 핵심 차별점이고, 데이터·점수·UI는 보조 격이다.
- **매일 트리거** — Vitamin이 아니라 미니 Painkiller. 직장인 대상 공감대 + 매일 캡쳐각이 누적된다.
- **무인증 데이터 단일 소스** — 4시간 제약에서 키 발급/CORS 디버깅을 0으로 만든다.
- **DB 없이 시연 가능** — localStorage만으로 누적 마일리지 가치를 보여줄 수 있어, 인증·스키마 비용을 회피한다.

## Key Assumptions to Validate

- [ ] **Must** — Open-Meteo air-quality + UV가 한국 주요 도시 좌표에서 결측 없이 응답한다. 검증: 서울·부산·대구·인천·대전 좌표 5개로 사전 fetch 테스트 (Plan 직전).
- [ ] **Must** — Gemini Flash가 같은 점수 범위라도 매번 다른 칭호/서사를 생성한다. 검증: 동일 입력 10회 호출 → 칭호 중복률 30% 이하인지 prompt 1차 튜닝.
- [ ] **Should** — 단일 카드 UI(점수·칭호·서사·날짜)가 "캡쳐하고 싶다" 첫인상을 만든다. 검증: Sketch 단계(wireframe.html)에서 동료 1명 30초 노출 후 반응.

## MVP Scope

**포함 (1~2 features, 2~3 pages, 5~7 scenarios)**
- 메인 페이지: 도시 선택 → 오늘의 점수 + 칭호 + 1줄 서사 카드
- 마일리지 페이지: 누적 점수 합계 + 일자별 칭호 캘린더 뷰
- 점수 산출 로직 (PM2.5/PM10/강수/풍속/UV/특보 가중 합)
- AI 호출 (Gemini Flash, server-side route handler 경유)
- localStorage 일자별 누적
- 공유 카드 (캡쳐 친화 카드 UI + 클립보드 복사)

**제외**
- 친구 랭킹·소셜
- 일러스트/이미지 자동 생성
- 푸시 알림·이메일
- 로그인·회원가입
- 동네 자동 위치 검색 (대신 한국 주요 도시 10~15개 드롭다운으로 단순화)

## Not Doing (and Why)

- **친구 랭킹** — 인증·DB·실시간 동기화 필요. 4시간 스코프 폭발.
- **일러스트 자동 생성** — 외부 이미지 API 의존, 비용/지연/실패 모드 셋이 동시에 들어옴. 카드 디자인(타이포·컬러·아이콘)으로 시각적 만족을 대체한다.
- **푸시 알림** — 사용자가 매일 들어오는 모델로 충분. 알림은 PWA·서버 사이드 cron이 필요해 4시간 안 됨.
- **동네/주소 검색** — 카카오 로컬 API 키 발급·인증·CORS 처리에 시간 소요. 도시 드롭다운으로 시연 가치 동일.
- **회원·인증** — localStorage가 단일 디바이스에서 충분히 누적 가치 시연.

## Open Questions

- **점수 가중치 공식 확정** — MVP 1차 안: `PM2.5 × 0.5 + PM10 × 0.3 + 강수mm × 1.5 + 풍속m/s × 1.0 + UV × 1.0 + 기상특보 +20`. spec 단계에서 표로 못 박는다.
- **칭호 다양성 확보 전략** — Gemini prompt에 `(1) 매일 어휘 풀을 다르게 (2) 점수 구간별 톤 다르게 (3) 같은 칭호 반복 금지`를 시스템 프롬프트에 명시할지, 후처리(클라이언트 dedup)로 처리할지 — 1차는 프롬프트에서.
- **캘린더 라이브러리** — shadcn calendar로 시작, 칭호 표시는 day cell custom render. Plan 단계에서 선택.
- **마일리지 = 합계 vs 레벨** — 1차는 단순 합계. 시간 남으면 레벨/배지 추가 (스트레치).
